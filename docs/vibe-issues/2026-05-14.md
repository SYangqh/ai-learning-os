### B1-01 | concept/practice/retro 节点每次 AI 回复后无条件推进，学员说"不知道"也往前走

**现象**：学员在 `concept` 节点回答"不知道"，AI 讲解后系统自动推进到 `practice`；学员再答"不知道"，自动推进到 `task`；到 `task` 节点没有代码，系统报错"请先提交你的代码或作品，才能进入评审阶段"——这个报错本身是正确的，但不应该让学员在没有理解概念时就到达 `task` 节点。

**根因**：`SessionService.persistReply()` 每次收到 AI 回复后无条件调用 `nextNode()` 推进，没有任何"学员是否真正理解"的判断逻辑。节点状态机的"推进"完全由服务端决定，而不是由教学语义决定。

**影响**：
- 学员在 `concept` 节点反复说"不知道" → 被强推进 `task` → 遇到门控报错 → 体验断裂
- `practice` 节点同理：答错了也往前推，任务到来时学员毫无准备
- 整个"必须学会才能过"的核心价值观名存实亡

**修复**：
1. 新增 `AI_ADVANCE_CONTROLLED_NODES = Set.of("concept", "practice", "retro")`
2. 更新这三个节点的 System Prompt，明确告诉 AI：
   - `concept`：学员说"不知道"或仍有疑问 → 继续讲解，**不加 `[ADVANCE]`**；学员展示出基本理解后加
   - `practice`：学员回答不正确 → 继续引导，**不加 `[ADVANCE]`**；回答正确才加
   - `retro`：总结完成后加 `[ADVANCE]`
3. `persistReply()` 检测 `[ADVANCE]` 标记：AI 受控节点未包含时停留当前节点，允许多轮对话
4. `[ADVANCE]` 在返回用户回复前剥离，不展示给学员

**防范规则**：任何新增节点类型，必须在设计阶段明确"推进由谁控制"：
- 服务端逻辑控制（artifact 提交门控 / Rubric 评审结果）→ 加入 `ARTIFACT_REQUIRED_NODES` 或 REVIEW 逻辑
- AI 教学判断控制（是否真正理解） → 加入 `AI_ADVANCE_CONTROLLED_NODES` + Prompt 指令
- 一轮即推进（引导性开场）→ 默认行为（intro 节点）

### B1-02 | task 节点门控的报错信息暴露了系统内部逻辑，用户体验差

**现象**：学员在没有准备好的情况下被推进到 `task` 节点，点击"提交"后收到 AI 回复："请先提交你的代码或作品，才能进入评审阶段。" 这条消息语义准确，但对于一个刚被推进到 `task`、完全没有写代码的学员来说，显得突兀且缺乏引导。

**根因**：门控拦截逻辑直接返回硬编码字符串，没有结合当前任务说明，也没有给出"你需要做什么"的具体引导。本质上是 B1-01 的下游症状——如果推进逻辑正确，学员不应该在没有概念理解的情况下出现在 `task` 节点。

**修复（当前）**：B1-01 修复后此问题将显著减少，学员只有在真正经过 `concept` + `practice` 后才进入 `task`。

**遗留优化（待做）**：门控消息应结合 Stage 的 `task_description` 内容，给出更具体的任务说明，而不是通用拦截文案。可在 `AdvanceContext.earlyReturn` 构建时注入 Skill YAML 中的 `task_description`。

### B1-03 | Windows 控制台日志中文乱码

**现象**：Spring Boot 运行时日志出现乱码（如 `AI 鏈嶅姟鏆傛椂涓嶅彲鐢紝璇风◢鍚庨噸璇?`），实际内容为"AI 服务暂时不可用，请稍后重试"。

**根因**：Windows 控制台默认编码 GBK（代码页 936），Spring Boot 默认以 UTF-8 字节写出日志，两者不匹配。

**修复**：新增 `src/main/resources/logback-spring.xml`，在 `ConsoleAppender` 的 `encoder` 中显式声明 `<charset>UTF-8</charset>`。

**防范规则**（已写入 `backend_basics.skill.yaml` pitfall #6）：项目初建时就加入 `logback-spring.xml`，避免后期排查日志时的干扰。

### B1-04 | API Key 校验失败页面显示 "Internal Server Error" 而不是可读错误

**现象**：用户选错服务商（如选 OpenAI 填了 DeepSeek key），提交后页面提示 "Internal Server Error"，无法自救。

**根因**：两层问题同时存在：
1. `testConnection` 把内部 `AppException` 的 message（"AI 服务暂时不可用"）直接拼到用户提示里，语义完全错误
2. 前端 `apiFetch` 用 `??` 提取 message：`json?.message ?? json?.msg ?? text`，当 message 为空字符串 `""` 时 `??` 不会跳过，fallback 失败，最终把原始 JSON 文本展示给用户

**修复**：
1. `DynamicChatService.testConnection`：不再拼接内部异常消息，改用固定中文提示 + `log.warn` 记录原始原因
2. `apiFetch`：`??` 改为 `||`，让空字符串也能 fallback 到 `json?.error` 字段

**防范规则**（已写入 `backend_basics.skill.yaml` pitfall #9）：禁止在任何场景向用户展示 "Internal Server Error"、"null"、原始 HTTP 状态码文本。

### B1-05 | inlineKey 降级路径 provider 识别不可靠，选错服务商静默路由到 OpenAI

**现象**：用户注册向导中手动选 OpenAI 但填入 DeepSeek key，系统将该 key 发到 `api.openai.com`，触发网络层错误，用户收到不可读的提示。

**根因**：`resolveContext()` 的 inlineKey 降级路径只有 `sk-ant-` → Anthropic，其余全走 OpenAI，不区分 DeepSeek / 通义千问等服务商。

**修复**：保存凭据前增加连通性测试步骤，彻底绕开前缀猜测：
1. 后端新增 `DynamicChatService.testConnection(providerKey, apiKey)` — 用传入 key 发送测试消息，失败抛用户友好异常
2. 后端新增 `POST /api/llm/credentials/test` 端点 — 不保存，只测试连通性
3. 前端 `handleApiKeySubmit` 改为先调 test 接口通过后才调 PUT 保存，强制要求用户先选服务商再输入 key
