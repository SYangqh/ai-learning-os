# Vibe Issues — 2026-05-15

## 本次实现：Phase 7 Artifact 类型可配置化

---

### 问题清单

#### #1 `Set` 未 import 导致编译失败
- **现象**：`ArtifactService.java` 添加 `VALID_TYPES = Set.of(...)` 后漏掉 `import java.util.Set;`，编译阶段才发现
- **根因**：手工追加常量时未检查 import 列表
- **修复**：补充 `import java.util.Set;`
- **改进建议**：每次新增集合类型常量，同步检查 import 是否已包含 `java.util.Set/Map/List`

#### #2 前端右侧面板提交按钮 disabled 逻辑需按类型分支
- **现象**：原代码 `disabled={!code.trim() || ...}` 仅判断 CODE 内容是否为空，NOTE/DIAGRAM 类型下即使有输入也无法启用按钮
- **根因**：代码是从 CODE-only 版本演化而来，disabled 条件没有跟随 artifactType 做条件分支
- **修复**：改为 `(artifactType === 'CODE' ? !code.trim() : artifactType === 'DIAGRAM' ? !diagramUrl.trim() : !noteContent.trim())`
- **改进建议**：产出面板每次扩展类型时，disabled/placeholder/submit-text 三处必须同步修改

#### #3 SkillRubricLoader 默认 artifact_type 为 CODE（向后兼容）
- **现象**：旧 Skill YAML 没有 `artifact_type` 字段时，`loadArtifactType` 应返回 "CODE" 而非 null/报错
- **根因**：设计时未考虑旧数据兼容
- **决策**：`loadArtifactType` 返回 `stage.artifact_type ?? "CODE"`，NONE 类型通过 `!"NONE".equals(type)` 跳过门控
- **改进建议**：新增字段时均需考虑旧记录的默认值语义

#### #4 前端历史记录"加载到编辑器"未覆盖 NOTE/ESSAY/PROOF 类型
- **现象**：历史提交列表中，只有 CODE 类型有"加载到编辑器"按钮，NOTE 类型提交后无法回填
- **根因**：历史区原始代码只处理了 CODE 分支
- **修复**：为 NOTE/ESSAY/PROOF 类型增加"加载到笔记区"按钮，调用 `setNoteContent(a.content)`
- **改进建议**：历史区的操作按钮必须与产出区的输入控件保持镜像对应
