## 问题清单

### P0 — 阻断编译的编码问题

#### 1. DynamicChatService.java BOM 字符导致级联编译失败

**现象**：`mvnw compile` 报 `非法字符: '\ufeff'` + 大量 `未结束的字符串文字`，同时 Lombok 注解处理失败，导致所有依赖 `@Slf4j`/`@Getter` 的文件报 `找不到符号 log`/`getId()` 等。

**根因**：上一轮会话中使用 PowerShell `Set-Content` 写入文件时采用了 UTF-8 with BOM 编码（Windows PowerShell 5.1 默认行为），Java 编译器不识别 BOM。

**修复**：用 `[System.IO.File]::WriteAllText(path, content, New-Object System.Text.UTF8Encoding $false)` 重写文件（显式指定无 BOM）。

**教训 → 规则**：凡用 PowerShell 写 Java/YAML/JSON 文件，必须用 `New-Object System.Text.UTF8Encoding $false`，绝不用 `Set-Content` 或 `Out-File` 默认模式。

---

#### 2. UserRateLimitInterceptor.java 双类定义 + 乱码中文字符串

**现象**：编译报 `类重复: UserRateLimitInterceptor`；截断旧类之后又报 `未结束的注释` / `未结束的字符串文字`。

**根因**：上一轮会话 `replace_string_in_file` 追加了新类定义到旧类定义之后，而不是替换。旧类注释和中文字符串均为 GBK 乱码（由 BOM 写入时的编码混乱导致）。

**修复**：同样用无 BOM UTF-8 完整重写文件，移除重复类。

---

### P1 — 设计问题

#### 3. Artifact 实体缺少 score/feedback 字段

**现象**：`ReportService` 初稿引用了 `artifact.getScore()` / `artifact.getFeedback()`，编译失败。

**根因**：Artifact 实体在 Phase 2 设计时只有 `status`，rubric 评审结果存放在 `LearningSession.progress` JSONB 里（`rubric_score`/`rubric_feedback`），没有冗余到 Artifact 上。

**修复**：ReportService 改为从 `session.getProgress()` 读取 rubric 结果，Artifact 只展示 `type` + `status` + `content` 预览。

**教训**：在引用实体字段前先用 `grep_search` 确认字段是否存在，不要凭记忆猜测。

---

#### 4. LearningSessionRepository 缺少 findByStageId

**现象**：ReportService 调用 `sessionRepository.findByStageId(stageId)`，Repository 接口未定义该方法。

**修复**：在 `LearningSessionRepository` 中添加 `List<LearningSession> findByStageIdOrderByStartedAtDesc(UUID stageId)`。

## 规划文档补充记录（本轮仅文档）

### 决策 1：不要把新需求塞回已完成的 Phase 6-8

**原因**：README 中 Phase 6-8 已标记完成。如果把“主题化反馈”“移动端 App”“混合作答模式”直接补进这些已完成阶段，会让完成定义失真。

**处理**：在 `ARCHITECTURE.md` 中新增独立的 `Phase 9：移动端与情绪化学习体验`，并单独补充 PRACTICE/QUIZ 交互规范。

### 决策 2：预制答案必须是可配置能力，而不是全局默认

**原因**：概念判断题和信心建立题适合预制答案，但儿童数学、精确计算、默写等任务不适合默认给选项。

**处理**：规划中明确 `FREE_INPUT_ONLY / PRESET_ONLY / HYBRID` 三种模式，并允许按题型、Skill、Stage、用户设置覆盖。

### 决策 3：反馈特效不是单纯 UI 装饰，而是主题系统的一部分

**原因**：用户明确要求可爱风和正式风的反馈感受不一致，因此特效、音效、文案语气必须共用同一份主题配置，而不是各自分散实现。

### 决策 4：自定义“我想学”不能只靠前端加一个输入框

**原因**：如果前端只是允许自由输入，但后端仍然只认静态 Skill YAML，那么用户输入的自定义目标最终还是会退化成“固定枚举”。

**处理**：规划中把 Skill 系统升级为“模板 Skill + 动态 Skill 实例”的混合模型；路径生成时先匹配模板，匹配不上再生成动态 Skill，并把结果快照绑定到学习路径。

### 决策 5：动态 Skill 必须持久化并版本化

**原因**：如果每次进入学习都重新临时生成 Skill，Stage 内容、Rubric 和路径结构都会漂移，学习过程无法复现，也无法稳定评审。

**处理**：新增 `GENERATED` Skill 来源和 `skillSnapshotId` 概念，要求动态 Skill 落库存档，并在路径创建时固定版本。

### 决策 6：下一阶段规划必须继续拆小，不能把 4 类能力塞进一个大 Phase

**原因**：主题化反馈、混合作答、移动端 App、自定义目标与动态 Skill 生成分别跨越前端交互、端能力、后端匹配和持久化边界。如果继续用“下一阶段”一行带过，后续 README 和验收口径会再次失真。

**处理**：将后续规划在 `ARCHITECTURE.md` 中拆分为 `Phase 9A/9B/9C` 与 `Phase 10A/10B/10C` 六个子阶段，每个子阶段分别定义实施任务和验收标准。

### 决策 7：Skill 测试标准必须基于当前真实 Skill 清单，而不是抽象描述

**原因**：当前仓库实际只有 3 个模板 Skill，且自动化测试主要覆盖节点状态机，没有覆盖 YAML 结构、学科差异、模板匹配和 NOTE 类型链路。仅写抽象“以后要测试 Skill”没有可执行价值。

**处理**：新增 `docs/SKILL_TESTING_STANDARD.md`，以 `backend_basics`、`english_spoken`、`marxist_philosophy` 作为基线样本，定义结构校验、单元测试、集成测试、手工验收和未来 Phase 9/10 的扩展测试口径。

### 决策 8：现有 Skill 的 `analogy_map` 覆盖率要纳入文档化风险，而不是在测试里默认视为已满足

**原因**：架构规则要求 Skill 支持 `frontend/hardware/finance/product/other` 五类背景，但当前 3 个 Skill 的 `analogy_map` 只覆盖前四类，没有 `other`。这是现状事实，文档必须把它显式暴露出来，避免后续误以为基线已完整。

**处理**：在 Skill 测试标准文档中把 `analogy_map` 覆盖率列为必查项，并将当前缺口记录为待补齐项，而不是静默跳过。

  }

| 问题 | 决策 | 理由 |
|------|------|------|
